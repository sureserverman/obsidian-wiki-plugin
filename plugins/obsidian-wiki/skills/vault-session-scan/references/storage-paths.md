# Session storage paths and formats per tool

This reference describes where each supported AI coding tool stores session/conversation
history and what format each uses. Read this once when scanning sessions across tools.

If a tool's storage format changes, update this file via `vault-schema-maintain` or
inline (it's a reference doc, not the schema itself).

---

## Claude Code

**Path pattern**:
```
~/.claude/projects/<encoded-cwd>/<session-uuid>.jsonl
```

`<encoded-cwd>` is the absolute path of the working directory at session start, with
every `/` replaced by `-` and a leading `-`. Example:

| Working dir | Encoded |
|---|---|
| `/home/user/dev/noob-skills` | `-home-user-dev-noob-skills` |
| `/home/user/dev/knowledge` | `-home-user-dev-knowledge` |

**File format**: JSONL — one JSON event per line. Event types include:
- `user` — user message
- `assistant` — assistant message (with content blocks: `text`, `tool_use`)
- `tool_result` — output of a tool call
- `summary` — auto-generated session summaries

Sessions can be very long (10–100MB). Stream-parse line-by-line; do not slurp.

**Filtering**:
- mtime sort: `ls -t ~/.claude/projects/<encoded-cwd>/*.jsonl`
- Content sample: read first 50 lines and last 50 lines for a quick gist; use the
  `summary` events for an autogenerated overview if present.

**MCP shortcut**: the `episodic-memory` plugin (if installed) provides search MCP
tools (`mcp__plugin_episodic-memory_episodic-memory__search`,
`mcp__plugin_episodic-memory_episodic-memory__read`,
`mcp__plugin_episodic-memory_episodic-memory__show`) that index Claude Code sessions.
Prefer these over raw JSONL parsing when available — they handle tokenization and
return ranked snippets.

## Codex (OpenAI Codex CLI)

**Path pattern**:
```
~/.codex/sessions/<YYYY>/<MM>/<DD>/rollout-<ISO-timestamp>-<uuid>.jsonl
~/.codex/archived_sessions/...                                          # older runs
```

Sessions are bucketed by day. Filename includes ISO timestamp (`2026-03-08T17-18-47`)
and a UUID.

**File format**: JSONL events. Schema differs from Claude Code but conceptually similar
(messages, tool calls, tool results). Stream-parse.

**Filtering**:
- Recent sessions: `find ~/.codex/sessions/2026/<MM> -name 'rollout-*.jsonl' -mtime -7`
- The directory tree is already date-organized so date filters are cheap.

**Other relevant dirs**:
- `~/.codex/memories/` — long-term memory store (separate from sessions)
- `~/.codex/log/` — runtime logs

## Cursor

Cursor stores session data in **two places**, with different value:

### 1. Per-project tool outputs (easy to parse)

```
~/.cursor/projects/<encoded-cwd>/agent-tools/<uuid>.txt
~/.cursor/projects/<encoded-cwd>/assets/...
```

`<encoded-cwd>` is similar to Claude Code's encoding: e.g., `home-user-dev-foo`
(no leading `-`).

The `agent-tools/*.txt` files contain captured tool outputs (build logs, command
results). They are plain text and easy to grep, but they are **outputs only** —
not full conversation transcripts. Useful for "what command produced this error" but
not for "what was discussed".

### 2. Workspace storage (full conversations, hard to parse)

```
~/.config/Cursor/User/workspaceStorage/<workspace-hash>/state.vscdb
~/.config/Cursor/User/workspaceStorage/<workspace-hash>/anysphere.cursor-retrieval/
```

The full chat history lives in **SQLite** (`state.vscdb`) under the `anysphere`
extension namespace. Schema is undocumented and changes between Cursor versions.

To extract:
```bash
sqlite3 ~/.config/Cursor/User/workspaceStorage/<hash>/state.vscdb \
  "SELECT key, value FROM ItemTable WHERE key LIKE '%cursor%';"
```

The values are JSON blobs. Decode case-by-case.

**Pragmatic recommendation**: prefer the agent-tools `.txt` files for routine scans.
Only dive into the SQLite when the user is specifically looking for a conversation
they remember.

## Gemini

**Path pattern**:
```
~/.gemini/history/<project-name>/
~/.gemini/history/<project-name>/.project_root        # marker file
```

The `history/` directory has one subdir per project the user has worked on. Inside,
the actual conversation files are version-dependent — recent Gemini versions store
sessions in proprietary formats and locations that change frequently.

Other relevant Gemini dirs:
- `~/.gemini/agents/` — agent configurations
- `~/.gemini/commands/` — slash command definitions
- `~/.gemini/skills/` — installed skills
- `~/.gemini/antigravity/` — Gemini Code Assist's "antigravity" workspace
  - `code_tracker/`, `knowledge/`, `brain/` — interesting subdirs that may contain
    session-derived knowledge

**Pragmatic recommendation**: Gemini's session storage is the least stable of the
five tools. Treat it as best-effort: list `history/<project>/` for files, read what
you can identify, and tell the user if you can't find structured conversation data.

If `~/.gemini/antigravity/knowledge/` or `~/.gemini/antigravity/brain/` contains
markdown files, those are often Gemini's own auto-summaries — high-value sources for
import.

## OpenCode

**Path pattern**:
```
~/.local/share/opencode/storage/project/<project-hash>.json
~/.local/share/opencode/storage/project/global.json
```

Each project gets one JSON file containing its session state. There is a `global.json`
for cross-project state. There is also `~/.local/share/opencode/storage/migration/`
for schema migration data.

**File format**: JSON (not JSONL). The structure is OpenCode-version-dependent. Read
the file, locate the messages/turns array, iterate.

OpenCode storage is typically small (tens of KB) compared to Claude Code (100s of MB),
because it stores less per session. This makes scanning fast but means individual
sessions carry less detail.

## Cross-tool conventions for the wiki

When importing a session into the vault, use this filename convention in
`raw/sessions/`:

```
raw/sessions/<tool>-<YYYY-MM-DD>-<short-id>.md
```

Where:
- `<tool>` ∈ `claude-code`, `codex`, `cursor`, `gemini`, `opencode`
- `<YYYY-MM-DD>` is the session start date
- `<short-id>` is the first 8 chars of the session UUID (or a hash of the path if no
  UUID exists)

Example: `raw/sessions/claude-code-2026-04-07-2b7b05df.md`

This makes the source filename self-describing and prevents filename collisions
across tools.

---

## Idempotency

A session is "already imported" if **either**:

1. A file exists at `raw/sessions/<tool>-<date>-<short-id>.md`, OR
2. Some wiki page's `sources:` frontmatter array contains that path.

When scanning, surface only sessions that are NOT already imported. This avoids
re-suggesting the same conversation every run.
