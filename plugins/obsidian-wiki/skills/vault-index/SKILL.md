# Vault Index

Generate `<vault>/index.md` — a machine-readable, human-scannable digest of every page
in the Obsidian vault. The index lists each page's title, path, tags, topic mentions,
one-line summary, and last-updated date. Other tools (notably the `vault-context`
plugin used from project repos) read this file to find vault pages relevant to whatever
they're working on, without having to grep the whole vault.

The index is **derived data**: it can always be regenerated from the vault's pages.
Hand-edits are pointless — the next run overwrites them.

## Preconditions

- Resolve the vault path. If the cwd is the vault, use that. Otherwise call
  `${CLAUDE_PLUGIN_ROOT}/scripts/resolve-vault.sh` (env var → `~/.config/obsidian-wiki/config.json`
  → fallback `<vault>`).
- `CLAUDE.md` and `log.md` must exist at the vault root. If not, the vault has not been
  bootstrapped — point the user at the obsidian-wiki README's bootstrap section.

## Walk the vault

Enumerate `.md` files under the six category directories:

- `Architecture/`
- `Gotchas/`
- `Patterns/`
- `Platforms/`
- `Projects/`
- `Technologies/`

**Skip** these even if they're under a category dir:

- `Home.md` (the hand-curated MOC)
- `index.md` (the file you're about to write — never index yourself)
- `CLAUDE.md`, `log.md` (vault metadata)
- Anything under `raw/`, `raw/sessions/`, `raw/assets/`, `.obsidian/`

If the vault uses a different layout (per the user's `CLAUDE.md`), enumerate whatever
category dirs the schema declares. Don't hardcode the six.

## Per-page extraction

For each page, read it **once** and pull:

| Field | Source | Fallback |
|---|---|---|
| `title` | YAML frontmatter `title:` | filename without `.md` |
| `path` | path relative to vault root | required |
| `tags` | YAML frontmatter `tags:` (list or comma-separated) | empty |
| `topics` | proper-noun tokens in body (capitalized, ≥4 chars, ≤30 unique per page) | empty |
| `summary` | first sentence of body, OR a `## TL;DR` block if present, OR the line right after the H1 | empty string |
| `updated` | YAML frontmatter `updated:` | file mtime as `YYYY-MM-DD` |

**Topic extraction is fuzzy by design.** Pull capitalized tokens that look like proper
nouns (tools, protocols, projects, places). Filter out: page-internal headings, common
English words that happen to be capitalized at sentence start, and tokens shorter than 4
characters. Cap at 30 topics per page so the index file stays bounded.

**Summary length is one line.** Strip markdown formatting. If the first sentence is
longer than 200 characters, truncate at the nearest word boundary and append `…`.

## Index file format

Write to `<vault>/index.md`. **Full rewrite** every run — never patch in place.

```markdown
# Vault Index

Auto-generated by /obsidian-wiki:index on YYYY-MM-DD. Do not hand-edit.
Re-run after ingesting new sources.

Vault: <vault path>
Pages indexed: <N>

## Architecture/

### [[Page Title]]
- path: Architecture/Page Title.md
- summary: <one-line summary>
- tags: tag1, tag2, tag3
- topics: Topic1, Topic2, Topic3
- updated: YYYY-MM-DD

### [[Another Page]]
- path: Architecture/Another Page.md
- ...

## Gotchas/

### [[...]]
...
```

Group by category dir. Inside each category, sort pages alphabetically by title for
stable diffs across runs.

If a field is empty, write the field anyway with an empty value (`tags: `). Downstream
parsers expect a fixed set of keys per entry.

## Idempotency

Compare the new content against the existing `<vault>/index.md` byte-for-byte before
writing:

- If the file does not exist, create it.
- If the file exists and differs, overwrite it and remember the count of pages whose
  blocks changed.
- If the file exists and is byte-identical, do not write at all (preserve mtime).

## Log append

Append a single entry to `<vault>/log.md`:

```
## [YYYY-MM-DD] index | regenerated (N pages, M changed)
```

Where `N` is total pages indexed and `M` is the count of pages whose blocks differed
from the previous index. If the file was unchanged, write `(N pages, no changes)`.

## Report

Tell the user, in a few lines:

- Vault path
- Total pages indexed
- Pages whose entries changed (or "no changes")
- The path of `<vault>/index.md`

If a downstream tool (like `vault-context` from a project repo) prompted this run,
mention that the index is now ready for it.

## Common pitfalls

- **Indexing `raw/` or `.obsidian/`.** Never. Those are not wiki pages.
- **Indexing `index.md` itself.** Skip it explicitly — re-indexing your own output is
  a fast way to thrash.
- **Fabricating tags.** Tags come from frontmatter only. If a page has no `tags:`,
  write an empty `tags:` line. Do not guess based on content.
- **Hardcoding the six category dirs** when the user's `CLAUDE.md` declares a different
  set. Read the schema first.
- **Patching `index.md` in place.** Always write the whole file. Stable sort makes the
  diff small.
- **Skipping the log append.** The log is the audit trail; index runs must be visible
  there alongside ingest, lint, and the rest.
- **Writing on every run even when nothing changed.** Skip the write if the new content
  is byte-identical, so the file's mtime tracks real changes.
